# Центр сертификации ПАП

Веб-платформа Центра сертификации специалистов по прикладному анализу поведения.
Она сопровождает полный путь пользователя: от регистрации и выбора цели до проверки
документов, учёта CEU и часов, экзамена, выпуска сертификата и публикации в реестре.

Актуальность описания: **19 июля 2026 года**.

Продакшен: [account.reestrpap.ru](https://account.reestrpap.ru)

## Возможности платформы

### Пользователь

- регистрация, вход и восстановление пароля;
- редактирование профиля, фотографии, описания и местоположения;
- выбор цели сертификации или ресертификации;
- активный цикл с зафиксированными требованиями для выбранного уровня;
- загрузка и история документов, включая данные прошлых циклов;
- создание CEU-заявок, мультикатегории и история текущего/старых циклов;
- учёт практики, супервизии и менторства;
- выбор проверяющих и работа с заявками на подтверждение часов;
- заявка на экзамен и просмотр состояния допуска;
- просмотр выданных сертификатов;
- направляющая «Что делать сейчас» на основе текущего состояния пользователя;
- запрос на архивирование собственного профиля.

### Проверяющий

- список кандидатов на супервизию и менторство;
- принятие или отклонение сотрудничества;
- просмотр истории заявок;
- проверка и подтверждение часов кандидата;
- сообщения от посетителей публичного реестра.

### Администратор

- единая панель рабочих задач и уведомлений;
- поиск, фильтрация, архивирование и восстановление пользователей;
- карточка пользователя с заметками и журналом административных действий;
- управление уровнем, ролью, видимостью в реестре и данными профиля;
- ручная корректировка CEU и часов активного цикла;
- проверка документов, CEU, часов и экзаменационных заявок;
- обработка заявок внешних супервизоров;
- ручное управление платежными статусами;
- выпуск, замена, редактирование и отзыв сертификатов;
- пользовательский баннер;
- выгрузка XLSX/CSV, экспорт карточки пользователя и резервная копия PostgreSQL.

### Публичная часть

- реестр специалистов и соискателей без авторизации;
- фильтры по имени и местоположению, карточный и списочный режимы;
- публичный профиль и проверка сертификата;
- форма обращения к сертифицированному специалисту.

## Роли, уровни и циклы

В проекте важно не смешивать три разных понятия.

### Системные роли

- `STUDENT` — обычный пользователь;
- `REVIEWER` — проверяющий;
- `ADMIN` — администратор.

Роль определяет права доступа, но не квалификацию.

### Квалификационные уровни

- Соискатель;
- Инструктор;
- Куратор;
- Супервизор;
- Опытный Супервизор.

Уровень хранится через группы и определяет доступные цели и требования.

### Сертификационные циклы

Цикл имеет тип `CERTIFICATION` или `RENEWAL`, целевой уровень и состояние
`ACTIVE`, `COMPLETED` или `ABANDONED`. CEU, часы, документы, экзаменационные
заявки и выданный сертификат связываются с конкретным циклом.

Публичный реестр использует сертификат как источник квалификационного статуса:

- действующий сертификат — вкладка «Специалисты», даже если идёт новый цикл;
- действующего сертификата нет, но есть активный цикл — вкладка «Соискатели»;
- нет ни сертификата, ни активного цикла — пользователь не попадает в эти вкладки;
- дополнительно учитываются архивирование и административная настройка видимости.

## Архитектура

```text
gc_project/
├─ frontend/                 React SPA
│  ├─ src/components/        общие UI-компоненты
│  ├─ src/features/          функциональные модули
│  ├─ src/pages/             страницы маршрутизации
│  ├─ src/utils/             форматирование и чистые расчёты
│  └─ deploy/nginx.conf      nginx контейнера frontend
├─ backend/                  Fastify API
│  ├─ handlers/              бизнес-операции
│  ├─ routes/                регистрация HTTP-маршрутов
│  ├─ middlewares/           аутентификация и доступ
│  ├─ prisma/                схема, миграции и seed
│  ├─ utils/                 расчёты, файлы и инфраструктурные функции
│  └─ server.ts              точка входа
└─ docker-compose.yml        PostgreSQL, backend и frontend
```

Frontend обращается к API через `VITE_API_URL`. В Docker запросы `/api/*` и
`/uploads/*` проксируются контейнером nginx на Fastify. PostgreSQL доступен через
Prisma. Пользовательские файлы лежат вне контейнера в постоянном томе/каталоге.

## Переменные окружения

Backend читает `backend/.env`. Минимальная конфигурация:

```dotenv
DATABASE_URL=postgresql://USER:PASSWORD@HOST:5432/DATABASE?schema=public
JWT_SECRET=replace-with-a-long-random-secret
RESET_PASSWORD_SECRET=replace-with-another-long-random-secret
FRONTEND_URL=http://localhost:5173
PORT=3000
```

Для отправки писем также используются:

```dotenv
SMTP_HOST=
SMTP_PORT=465
SMTP_SECURE=1
SMTP_USER=
SMTP_PASS=
SMTP_DEBUG=0
```

Опционально:

- `UPLOAD_DIR` — корень файлового хранилища. Без него локальный backend использует
  `frontend/public/uploads`;
- `UPLOAD_MIN_FREE_MB` — минимальный свободный объём upload-хранилища для healthcheck
  (по умолчанию 256 МБ);
- `LOG_LEVEL` — уровень Pino (`debug` локально, `info` в production по умолчанию);
- `SENTRY_DSN`, `SENTRY_ENVIRONMENT`, `SENTRY_RELEASE` — backend-диагностика ошибок;
- `NODE_ENV` — в production включает обязательность `JWT_SECRET` и публичное
  минутное кеширование реестра.

Frontend использует:

```dotenv
VITE_API_URL=http://localhost:3000
```

Для frontend-диагностики опционально используются `VITE_SENTRY_DSN`,
`VITE_SENTRY_ENVIRONMENT` и `VITE_SENTRY_RELEASE`.

В Docker production-сборка получает `VITE_API_URL=/api` автоматически.

Не добавляйте `.env`, дампы БД и пользовательские загрузки в Git.

## Локальный запуск

Потребуются Node.js 20+, npm и доступный PostgreSQL. Для генерации превью
сертификатов дополнительно нужен `pdftoppm` из Poppler.

### Backend

```bash
cd backend
npm ci
npx prisma generate
npx prisma migrate deploy
npm run dev
```

API будет доступен на `http://localhost:3000`. Проверка процесса:

```text
GET http://localhost:3000/
→ { "ok": true }
```

Полная проверка БД, upload-хранилища и фоновой задачи доступна через
`GET http://localhost:3000/health`.

Для пустой локальной базы при необходимости можно выполнить:

```bash
npx prisma db seed
```

Не запускайте seed на production без отдельной проверки содержимого скрипта.

### Frontend

В отдельном терминале:

```bash
cd frontend
npm ci
npm run dev
```

Vite по умолчанию поднимает приложение на `http://localhost:5173`.

## Запуск через Docker

Перед первым запуском создайте `backend/.env`. В `DATABASE_URL` имя хоста
PostgreSQL внутри Compose — `db`, а учётные данные должны совпадать с настройками
сервиса `db` в `docker-compose.yml`.

```bash
docker compose up --build -d
docker compose ps
docker compose logs --tail=100 backend frontend db
```

Backend при старте автоматически выполняет `prisma migrate deploy`. Файлы
production-конфигурации монтируются в `/app/uploads`; удалять этот каталог при
пересборке нельзя.

## Файлы и PDF-превью

- общий предел размера загружаемого файла — 10 МБ;
- документы отправляются пачками не более 10 файлов;
- исторические документы других циклов не расходуют лимит текущей отправки;
- сертификат принимается только при наличии настоящей PDF-сигнатуры;
- превью сертификата создаётся backend через `pdftoppm` и отдаётся браузеру как PNG;
- сгенерированные превью хранятся в `.previews/certificates` внутри файлового
  хранилища и пересоздаются при замене сертификата.

## Production-деплой

Код приложения запекается в Docker-образы, поэтому одного перезапуска контейнеров
после `git pull` недостаточно. Обычная последовательность:

```bash
cd /srv/cspap
git pull --ff-only
docker compose build backend frontend
docker compose up -d --no-build
docker compose ps
docker compose logs --tail=100 backend frontend
```

Эквивалентная короткая команда:

```bash
docker compose up --build -d
```

Перед крупным обновлением обязательно зафиксируйте текущий commit, сделайте dump
PostgreSQL и сохраните/пометьте текущие Docker-образы. После запуска проверьте:

- `/`, `/login`, `/registry`;
- `/api/registry`;
- авторизацию и личный кабинет;
- загрузку файла;
- открытие сертификата и его PNG-превью;
- отсутствие новых ошибок в логах backend/frontend.

Откат кода не откатывает уже применённые миграции. Если старая версия приложения
несовместима с новой схемой, нужен заранее подготовленный dump БД и отдельное окно
обслуживания.

## Важные эксплуатационные замечания

Подробная памятка по healthcheck, логам, Sentry и командам диагностики находится
в [`docs/monitoring.md`](docs/monitoring.md).

- `docker system prune -a` удаляет все неиспользуемые образы, включая потенциальные
  rollback-образы. Выполняйте очистку только после проверки релиза и резервной копии;
- каталог загрузок и PostgreSQL volume являются постоянными данными и не должны
  удаляться вместе с контейнерами;
- endpoint резервного копирования требует установленный `pg_dump`; он включён в
  backend Docker-образ;
- production использует внешний HTTPS reverse proxy, а контейнеры frontend/backend
  публикуются наружу через серверный `docker-compose.override.yml`;
- npm-зависимости в Docker собираются через настраиваемое зеркало, потому что на
  сервере возможны таймауты основного npm registry.

## Основные API-направления

- `/auth` — вход, регистрация и восстановление пароля;
- `/ceu` и `/admin/ceu` — CEU и административный аудит;
- `/supervision` и `/admin/supervision` — часы, связи и проверка;
- `/document-review-request` и `/document-review-requests` — документы;
- `/exam-applications` — экзаменационные заявки;
- `/certificates` и `/admin/certificates` — сертификаты и превью;
- `/payment` — платежные состояния;
- `/registry` — публичный реестр;
- `/notifications` — уведомления;
- `/upload` и `/uploads` — файловое хранилище;
- `/admin/users` — пользователи и административная карточка;
- `/specialist-contact-messages` — обращения к специалистам;
- `/admin/db/backup` — резервная копия PostgreSQL.

Точные маршруты и правила доступа следует проверять в `backend/routes` и
`frontend/src/App.tsx`: код является источником истины.
